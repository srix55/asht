                                Users #

                                Types of users :-

                                • Super-administrators
                                • Others

                                Teams #

                                A collection of users with common roles (role + space) such that a new user assigned to a particular team gets a default team role. A user can belong to multiple teams. In case of conflict in permissions - suppose the user is restricted in a particular space but is assigned into a team that grants it, then the restriction takes priority.

                                Super-administrators #

                                Every client account has one or more super-administrators. By default, the one who creates an account with us is given the super-administrator status. Super administrators can do everything that’s permissible within the bounds of a client account. However, they cannot demote another super-administrator, even though they can promote users to super-administrators.

                                Users & Spaces #

                                • Every user is assigned a primary space, which he has default access to
                                • However, it can be changed to allow a user to have specific access rights (such as read-only, write-only) to specific one or more spaces

                                Spaces #

                                Spaces are used to group resources for better management & to separate concerns. Example, East-brewary organization has branch A & branch B spaces with their own players, content creators & billing cycles. Spaces are enabled in the app-store.

                                • Can be categorized in the UI as Organization/Branch/Space/Area/Zone
                                • Spaces can be active or suspended (owing to non payment of bills or misuse for example)
                                • Every resource (player, user, playlist, media) belongs to a space.
                                • A space is a tree structure with content-access optionally shared with descendents/ancestors of that space. For example, Let’s say an organization X has 3 branches A, B & C with C having indoor & outdoor areas. Then, Space X has three sub-spaces A,B & C with C having 2 sub-spaces indoor & outdoor.
                                • Content shared with ancestors/descendents can be of particular types only like media only, playlists only etc
                                • Access can also be given to particular spaces including or not including sub-spaces
                                • A space has a settings screen of its own
                                • It can have a setting that disables showing that a user doesn’t have a particular permission in PS
                                • A space can limit
                                  • Number of players under that space, exceeding which sends an alert or which cannot be exceeded
                                  • A fixed cost, exceeding which sends an alert or which cannot be exceeded

                                Security #

                                Every resource in the app such as player, playlist, media etc & actions on them (read/write) needs authorization. It’s managed with

                                • Permissions (or restrictions)
                                • Roles (collection of permissions/restrictions)
                                • Spaces (scope of security, app-optional)
                                • Resources (individually lockable entities, app-optional)
                                • Teams (collection of roles/permissions, app-optional)

                                What’s a resource? #

                                A resource can be specific media (pic, video, audio, app-specific like twitter-feed-of-so etc), playlist, folder, player, schedule. Each can be differentiated by its id. For example, pl-123, fo-123, pa-123, sc-123 etc

                                Permissions #

                                Permissions by themselves do not specify the resource, space scope, role, user or expiry dates. However they can be added to a user, team or role to include a specific resource, space or user with expiry dates. Examples:-

                                • player:get
                                • player:update
                                • player:create
                                • player.notes:create
                                • player.notes:delete

                                Roles & Permissions #

                                A role is a set of specific permissions (granted or denied) associated with specific resources (space is a resource too) and can have expiry dates. A role may be visible only in particular spaces, with optional subspaces. Requirements:-

                                • Permissions can not just be granted, but can also be denied. For example, user A does NOT have access to folder T.
                                • A user can have different roles in different spaces. Example, user A is a publisher in space first-floor but just a viewer in space london-branch.
                                • A permission in a role can have validity time period. Example, viewer role can see folder T only on 1st Jan 2024
                                • A permission in a role can be space-wide. Example, viewer role can see all folders of space T.
                                • Role visibility is a thing. Example, role first-floor-publisher is only visible in space first-floor and its subspaces

                                Permission Conflicts & Hierarchy #

                                • Direct user permissions (grant/deny) trumps role & team permissions
                                • Even in direct user permissions, granularity is given priority. For example,
                                  • if a user has playlist:set permission (grant) on a folder but a playlist:set permission (deny) on a particular playlist in that folder, then the deny wins.
                                  • Similarly, if playlist:set is granted at the space level but the folder denies it, the deny wins.
                                • Role permissions trump team permissions
                                • Even in role permissions, granularity matters. For example,
                                  • If the role grants playlist:set on the space but denies the same for a folder in that space, the deny wins.

                                Preset Roles #

                                • Viewer: Can see every part of the space he belongs to, except billing
                                • Content Maker: Can create, edit or delete media of the space he belongs to
                                • Admin: Can see & edit every part of the space he belongs to

                                Roles & Teams #

                                Teams can be enabled via the app-store. It’s just to make user-role management easier. Example, user A belongs to team content-makers-for-london-branch which is a team that’s given view & publish permissions in london-branch. A team can have validity dates, so can the roles that it constitutes. Team visibility is a thing, including option to make it visible in subspaces.

                                Permissions & Spaces #

                                • A space can be set to have its content be visible for its ancestors and/or descendants
                                • A space can also be set to have its content be visible in specific spaces & subspaces
                                • A particular resource of a space can be made visible for specific spaces & subspaces
                                • The space hierarchy is maintained as a materialized path (or enumerated path or lineage) in the database

                                Archived Ideas #

                                • Temporary permissions: A user gets to request for permissions to a resource upon which the administrator gets a notification to grant the permission for say 15 minutes. Though sounds great, it’s a lot of work and isn’t something anyone’s wanting to have. None of the competition has this.
                                • Permissions with expiry dates: An admin can grant permissions that are time bound with start and end date times. Not sure if anyone is asking for this. Not sure if it’s worth it - implementing.
                                • Teams with overridable permissions: A team’s collection of permissions (roles) can be overridden. Too many levels of flexibility leads to confusion as it can already be done at the role level.
                                • Soft deletes in the database. No need I guess. Have good auditing.

                                Settings #

                                • Organization-wide Settings: These are client-wide settings and apply to all spaces and sub-spaces unless spaces or sub-spaces supersede the org-wide settings.
                                • Space-specific Settings: These are specific to the spaces. Any subspaces created shall inherit the settings of its parent only at the time of creation. When parent’s settings are updated, the child’s won’t get updated automatically, unless the setting explicitly specifies so.

                                Organization-wide Settings #

                                • Language: A client only needs one or 2-3 languages for a specific location. When the languages are set organization-wide, only those are shown for both language-input and language-interface (save/cancel buttons, options, dashboard menu etc etc)
